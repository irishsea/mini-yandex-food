# Mini Yandex Food

Демонстрационная микросервисная платформа, моделирующая ключевые процессы сервиса доставки еды.

Проект построен как исследование архитектуры распределенных систем: взаимодействие сервисов через Kafka, работа с согласованностью данных, обработка заказов и событийная модель домена.

---

## Цель проекта

Основная цель - проработка типичных задач backend-архитектуры:

- декомпозиция домена на сервисы
- асинхронное взаимодействие между компонентами
- устойчивость к сбоям и повторной доставке событий
- масштабирование обработки заказов

---

## Архитектура системы

Система построена вокруг событийной модели:


Order Service → Kafka → Delivery Service / Payment Service


Каждый сервис является автономной единицей со своей базой данных и отвечает за отдельный контекст.

---

## Почему Kafka

Kafka используется как основной механизм интеграции между сервисами по следующим причинам:

- **Развязка сервисов**: producer не зависит от состояния consumer'ов
- **Асинхронность**: обработка заказов не блокирует основной поток выполнения
- **Масштабируемость**: consumers можно горизонтально масштабировать через consumer groups
- **Устойчивость к сбоям**: события сохраняются и могут быть переобработаны

Kafka здесь - слой событийной интеграции между доменами.

---


## Масштабирование

Архитектура изначально рассчитана на горизонтальное масштабирование:

- Order Service - масштабируется по входящим запросам
- Kafka consumers - масштабируются через consumer groups
- Stateless-сервисы позволяют легко увеличивать количество инстансов

Узкие места устраняются через:
- разделение топиков по доменам
- партиционирование событий
- изоляцию БД на сервис

---

## Архитектурные решения

### 1. Одна БД на сервис
Каждый сервис владеет своей базой данных, исключая переиспользование между доменами.

### 2. Событийная коммуникация
Интеграция реализована через события, а не синхронные REST-вызовы.

### 3. Идемпотентность
Обработка событий построена с учетом возможных повторных доставок сообщений.

### 4. Разделение ответственности
Каждый сервис отвечает строго за свою часть домена без перекрытия ответственности.
